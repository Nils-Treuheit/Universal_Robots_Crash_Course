# High-Level Behavior

The robot maintains a virtual position:

```python
current_pos
```

and a state:

```python
grabbedCube = True/False
```

The user interacts by pushing on the TCP.

| Force Direction | Action                  |
| --------------- | ----------------------- |
| Z- (< -18N)     | Pick or Place           |
| X+ (> 18N)      | Take or Give            |
| Y- (< -18N)     | Move target 75 mm left  |
| Y+ (> 18N)      | Move target 75 mm right |

So the robot is basically a force-controlled state machine.

# State Machine

## State 1: No Cube

```python
grabbedCube == False
```

### Push Down (Z-)

```python
Pick()
```

Robot:

1. Goes to top_pick
2. Opens gripper
3. Descends until contact
4. Opens wider
5. Checks TCP height
6. If object exists:

   * descend 27 mm
   * close gripper
   * grabbedCube=True
7. Return to top_pick

Essentially:

```text
Pick object from surface
```

---

### Push Right (X+)

```python
TakeCube()
```

Robot:

```text
Move to handover position
Open gripper
Wait until user presses object into gripper
Close gripper
Return
```

This is a human-to-robot transfer.

---

## State 2: Cube Held

```python
grabbedCube == True
```

### Push Down (Z-)

```python
Place()
```

Robot:

```text
Move above place position
Descend until contact
Open gripper
Retreat
```

Classic contact-based placement.

---

### Push Right (X+)

```python
GiveCube()
```

Robot:

```text
Move to handover position
Wait until user pulls/pushes with enough force
Open gripper
Return
```

Robot-to-human transfer.

# What I Would Reduce It To

Conceptually this whole script is:

```python
state = EMPTY or HOLDING

while True:

    force = get_tcp_force()

    if force.z < -18:

        if state == EMPTY:
            pick()
        else:
            place()

    elif force.x > 18:

        if state == EMPTY:
            take_from_human()
        else:
            give_to_human()

    elif force.y < -18:

        shift_target(-75mm)

    elif force.y > 18:

        shift_target(+75mm)
```

That's the entire application logic in about 20 lines.

# My interpretation of the application

It appears to be a **collaborative cube sorting / handoff demo**:

* Human nudges robot in Y → select a position.
* Push down → pick/place at selected position.
* Push sideways → exchange object with human.
* Robot remembers whether it's holding something via `grabbedCube`.
* Contact detection is used instead of fixed heights to find surfaces.

The actual behavior is fairly elegant; most of the 1300+ lines are URCap-generated scaffolding. The core logic is probably under 150 lines and could realistically be rewritten into a clean hand-written URScript of roughly 100–200 lines with helper functions for:

* force command decoding,
* move-until-contact,
* gripper open/close,
* orientation handling,
* state management.

# 9. Force-trigger logic may retrigger endlessly

Main loop:

```python

```

or

```python

```

etc.

After completing an action:


you immediately return to:


and read force again.

If the operator is still pushing in that direction, you'll retrigger.

Usually I'd expect:


before accepting another command.
